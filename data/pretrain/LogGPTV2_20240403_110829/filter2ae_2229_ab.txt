### PyPI投放evilpackage的效果
从最近几天的数据来看，在PyPI上投放`evilpackage`的效果非常显著，每天大约有2200多个独立IP进行访问。这一数据量略高于之前的一位开发者所记录的数据，表明即使在类似的文章发布后一年，用户随意安装包的情况不仅没有改善，反而可能更加严重。

在那位开发者释放所有包之后，我作为一个“恶意者”再次使用他之前使用的`git`和`openssl`等名字来统计数据。最终，我也被官方要求删除所有包，并重新释放这些包名。我很好奇下一个“恶意者”会是谁，以及他们会在这类包中放入什么内容，是像我一样收集数据，还是执行更危险的操作如`rm -rf /`或勒索。

### `requirements.txt`作为恶意包的名字
经常使用Python的人通常知道`requirements.txt`是项目的依赖文件，通过以下命令安装：
```bash
pip install -r requirements.txt
```
但有时用户可能会误输入为：
```bash
pip install requirements.txt
```
因此，`requirements.txt`也是一个潜在的恶意包名。

#### 诡异的`requirements.txt`
2017年5月24日晚上，我尝试在PyPI上注册并上传`requirements.txt`，但失败了。然而，[GitHub Issues](https://github.com/search?utf8=%E2%9C%93&q=repo%3Afate0%2Fcookiecutter-evil-pypackage+requirements.txt&type=Issues)上竟然有153个与`requirements.txt`相关的Issues。尽管我在本地尝试安装也失败了，且PyPI上确实不存在`requirements.txt`信息，但我仍不清楚这种情况为何发生。

#### 绕过PyPI对`requirements.txt`的限制
在PyPI账号被删除后，我对`requirements.txt`仍然充满好奇。通过查看PyPI的源代码，我发现PyPI直接硬编码禁止上传`requirements.txt`、`rrequirements.txt`、`requirements-txt`和`rrequirements-txt`。进一步研究发现，可以通过正则表达式绕过这一限制，例如上传`requiremnets--txt`。最终，通过打包成wheel格式，成功绕过了这一限制，并通过添加一个名为`ztz`的source package来提醒用户。

### 总结
我是如何被PyPI官方发现的？原因很简单，每个恶意包都使用同一个cookiecutter模板生成，且描述均为`just for fun : )`，导致在PyPI首页出现了一排这样的描述。为了避免被发现，可以为每个恶意包准备不同的模板、文档和PyPI账号，并使用不同IP上传。然而，防御措施应主要依靠提升用户的信息安全意识，对待PyPI上的不熟悉项目应持不可信态度，意识到随意`pip install`就像随意执行`exe`一样危险。
