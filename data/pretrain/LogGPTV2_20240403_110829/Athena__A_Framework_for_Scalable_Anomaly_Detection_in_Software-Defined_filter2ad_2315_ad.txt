### 优化后的文本

多个研究项目 [26], [6], [9], [5], [39] 成功地利用了通过监控 OpenFlow 协议获取的基于流量的网络特征，来检测诸如 DDoS 攻击和交换机异常等网络异常。Mehdi 等人 [8] 探讨了将传统的异常检测技术与 OpenFlow 的监控能力相结合的可行性，而 Braga 等人 [10] 则利用 OpenFlow 统计信息展示了低成本的 DDoS 洪水攻击检测。

FRESCO [37] 提供了一个基于脚本的开发环境，有助于创建能够监控知名网络特征（如 TCP 会话）的安全应用程序。然而，这些努力并未充分考虑如何充分利用 SDN 环境中派生的网络特征来监测行为和操作稳定性。

从检测策略的角度来看，大多数相关工作 [26], [6], [9], [5], [39], [8], [10] 都集中在针对特定攻击的固定检测算法上，而不是通用目的的算法。尽管 FRESCO [37] 和 ATLANTIC [39] 提供了一系列旨在促进攻击检测的库（例如端口扫描），但只有前者提供了结构化的缓解措施，并且不支持多实例控制器环境。

可扩展的 SDN 监控：一些先前的项目探索了减少收集基于流量的网络特征开销的各种方法 [20], [19], [7], [3], [2], [4], [18]。实际上，已有项目采用了分布式架构，允许运营商大规模地收集网络特征 [17], [2], [18]。然而，这些项目并未提出跟踪恶意网络行为的检测策略。此外，大多数这些项目假设使用额外的定制交换机来进行可扩展的网络监控 [17], [18], [4], [3], [2]。

Bohatei [16] 提出了一种可扩展的 DDoS 检测和缓解方案，利用优化技术将任务分配给 NFV 机器以增加数据处理吞吐量。Bohatei 使用额外的 NFV 设备执行网络功能，但并不直接实现异常检测算法。还有一些基于采样的 SDN 监控方法，用于在从 OpenFlow 环境收集统计数据时减少采集开销 [9], [5]。这些技术依赖于 sFlow 采样。Spiffy [26] 展示了链路洪泛攻击缓解策略 [25]，利用 SDN 的集中管理插入流量缓解措施。Spiffy 的一个限制是需要定制交换机来测量网络行为。最后，还有一些项目探讨了在 SDN 中进行异常检测的可行性 [37], [10], [8], [6], [39]。与 Athena 不同，这些项目均未满足我们的可扩展性要求。

提高 SDN 可用性：一些先前的项目引入了北向 API，使运营商能够进行各种形式的网络监控 [42], [41], [2], [3], [4]。Payless [40] 提供了语义监控功能，帮助运营商通过调用结构良好的 RESTful API 来监控网络，这些 API 反映了高层次的监控需求。DNA [17] 引入了可扩展的网络监控功能，以检查遥测数据源。尽管这些先前的项目通过导出结构良好的 API 增强了可用性，但它们并未提供用于发现网络异常的检测算法。

### 结论
我们探讨了在大规模 SDN 环境中设计可扩展异常检测服务所面临的几个挑战。我们评估了我们在开源 ONOS 分布式 SDN 控制器上的解决方案原型 Athena。我们讨论了 Athena 如何通过其 API 抽象层使安全研究人员和开发人员能够以最小的编程努力制作异常检测应用程序。我们还讨论了通用的现成策略在驱动网络异常检测算法开发中的应用，并引入了一种新的 SDN 特定异常。Athena 采用分布式数据库和集群计算平台，可以在大规模分布式控制平面上部署这些检测算法。Athena 框架设计为在现有 SDN 基础设施上运行，使运营商能够以成本效益的方式部署它。我们的评估表明，Athena 可以高效地支持大规模数据中心类物理网络环境中的已知网络异常检测服务。Athena 已作为开源项目向学术界和 SDN 研究社区公开发布。
