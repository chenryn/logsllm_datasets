**DevOps 时代的 7 个领导力准则**

> DevOps 是一种持续性的改变和提高：那么也准备改变你所珍视的领导力准则吧。

![](/data/attachment/album/201808/02/232532zbdxg9kqi9kzqdd9.png)

如果 DevOps 最终更多的是一种文化而非某种技术或平台，那么请记住：它没有终点线。而是一种持续性的改变和提高——而且最高管理层并不及格。然而，如果期望 DevOps 能够帮助获得更多的成果，领导者需要修订他们的一些传统方法。让我们考虑在 DevOps 时代更有效的 IT 领导的 7 个想法。

### 1. 接受失败

“失败”这个词在 IT 领域中一直包含着非常具体的意义，通常是糟糕的意思：服务器故障、备份失败、硬盘驱动器损坏等。然而，在一个健康的 DevOps 文化中，失败被重新定义为机会。

Datical 的首席技术官兼联合创始人罗伯特·里夫斯表示：“在 DevOps 之前，我们有一种惩罚失败者的文化。我们学到的仅仅是避免错误。在 IT 领域避免错误的首要措施就是不要改变任何东西：不要加速版本迭代的日程，不要迁移到云中，不要做任何不同的事。”

这种旧时代的剧本已经不起作用了。事实上，停滞实际上是失败。“那些缓慢发布周期并逃避云的公司被恐惧所麻痹——他们将会走向失败，”里夫斯说道。“IT 领导者必须接受失败，并将其视为一个机遇。人们不仅从自己的过错中学习，也会从别人的错误中学习。开放和安全心理的文化促进学习和提高。”

**[相关文章：为什么敏捷领导者谈论‘失败’必须超越其本义](https://enterprisersproject.com/article/2017/10/why-agile-leaders-must-move-beyond-talking-about-failure?sc_cid=70160000000h0aXAAQ)**

### 2. 在管理层渗透 DevOps 理念

尽管 DevOps 文化可以在各个方向有机地发展，但正在从单体、孤立的 IT 实践中转变出来的公司可能需要高管层的全面支持。如果没有这种支持，信息会变得模糊，可能会鼓励那些宁愿按部就班的人。改变文化是困难的；人们需要看到高管层完全投入进去并且知道改变已经实际发生。

Rainforest QA 的首席信息官德里克·蔡指出：“高层管理必须全力支持 DevOps，才能成功实现收益。”成为 DevOps 商店涉及公司的方方面面，从技术团队到工具、流程、规则和责任。“没有高层管理的统一赞助支持，DevOps 的实施将很难成功。因此，在转变到 DevOps 之前在高层中保持一致是很重要的。”

### 3. 不只是声明 DevOps，还要明确其含义

即使 IT 公司已经开始拥抱 DevOps，也可能不是每个人都在同一个步调上。造成这种脱节的一个根本原因是人们对这个术语有不同的理解。

德里克解释道：“DevOps 对不同的人可能意味着不同的含义。对高管层和副总裁层来说，要执行明确的 DevOps 目标，清楚地声明期望的成果，充分理解这些成果将如何使公司的商业受益，并且能够衡量和报告成功的过程。”

事实上，在基线定义和远景之外，DevOps 要求进行频繁的交流，不仅是小团队内部，而是贯穿整个组织。IT 领导者必须将此设置为优先事项。“不可避免地，将会有一些阻碍，在商业中将会存在失败和破坏，”德里克说道，“领导者们需要清楚地向公司的其他人阐述这个过程，告诉他们作为这个过程的一部分可以期待的结果。”

### 4. DevOps 对商业和技术同样重要

成功的 IT 领导者将 DevOps 文化和实践视为一项商业策略，而不仅仅是构建和运营软件的方法。DevOps 是将 IT 从支持部门转变为战略部门的推动力。

CYBRIC 的首席技术官和联合创始人迈克·凯尔表示：“IT 领导者们必须转变他们的思想和方法，从成本和服务中心转变为驱动商业成果。DevOps 文化通过自动化和强大的协作加速这些成果。”

事实上，这是一个强烈趋势，贯穿这些新“规则”，在 DevOps 时代走在前沿。“促进创新并鼓励团队成员聪明地冒险是 DevOps 文化的关键部分，IT 领导者们需要在一个持续的基础上与他们沟通，”凯尔说道。

West Monroe Partners 的性能服务部门主任埃文补充道：“高效的 IT 领导者需要比以往任何时候都更积极地参与业务。每年或季度回顾的日子一去不复返了——你需要欢迎每两周一次的挤压整理，你需要有在年度水平上的思考战略能力，在冲刺阶段的互动能力，并在商业期望满足时给予奖励。”

### 5. 改变妨碍 DevOps 目标的任何事情

虽然 DevOps 的老兵们普遍认为 DevOps 更多的是一种文化而不是技术，但成功取决于通过正确的流程和工具激活文化。如果你声称自己是一个 DevOps 商店却拒绝对流程或技术做必要的改变，这就像是买了辆法拉利却使用了用了 20 年的引擎，每次转动钥匙都会冒烟。

例如，自动化是 DevOps 成功的重要并行策略。“IT 领导者需要重点强调自动化，”卡伦德说，“这将是 DevOps 的前期投资，但如果没有它，DevOps 将很容易被低效吞噬，无法完整交付。”

自动化是基石，但改变不止于此。“领导者们需要推动自动化、监控和持续交付过程。这意味着对现有实践、流程、团队架构以及规则的许多改变，”德里克说。“领导者们需要改变一切阻碍团队实现完全自动化的因素。”

### 6. 重新思考团队架构和能力指标

当你想改变时……如果你桌面上的组织结构图和过去大部分时候嵌入的名字都是一样的，那么是时候考虑改革了。

“在这个 DevOps 的新时代文化中，IT 执行者需要采取全新的方法来组织架构，”凯尔说。“消除组织边界限制，允许团队自我组织、敏捷管理。”

凯尔还建议在 DevOps 时代，这种反思也应该扩展到其他领域，包括你如何衡量个人或团队的成功，甚至是你与人们的互动方式。“根据业务成果和总体的积极影响来衡量主动性，”凯尔建议。“最后，我认为管理中最重要的一个方面是：同理心。”

注意，容易收集到的测量值并不是 DevOps 的真正指标。Red Hat 的技术专家戈登·哈夫写道：“DevOps 应该把指标以某种形式和商业成果绑定在一起。你可能并不真正在乎开发者写了多少代码，是否有一台服务器在深夜硬件损坏，或者你的测试有多么全面。你甚至都不直接关注你的网站响应情况或是更新速度。但你要注意的是这些指标可能与顾客放弃购物车转投竞争对手有关。”

### 7. 摒弃传统智慧

如果 DevOps 时代要求关于 IT 领导能力的新思考方式，那么也就意味着一些旧的方法要被淘汰。

“说实话，是全部，”凯尔说道，“摆脱‘因为我们一直都是以这种方法做事的’的心态。过渡到 DevOps 文化是一种彻底的思维模式转变，不是对瀑布式的过去和变革委员会的一些细微改变。”

事实上，IT 领导者认识到真正的变革要求的不只是对旧方法的小修小补。更多的是要求对之前的流程或策略进行全面重启。

West Monroe Partners 的卡伦德分享了一个阻碍 DevOps 的领导力的例子：未能拥抱 IT 混合模型和现代基础设施，如容器和微服务。“我所看到的一个大的规则是架构整合，或者认为在一个同质的环境下长期维护会更便宜，”卡伦德说。
