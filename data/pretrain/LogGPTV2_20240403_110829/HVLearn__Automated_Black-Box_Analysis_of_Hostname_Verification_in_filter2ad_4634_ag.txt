### SSL/TLS Libraries and Applications: Hostname Verification

#### 1. MatrixSSL
- **Supported Identifiers**: DNS, IPv4, and email.
- **Current Limitations**: The library does not yet support IPv6.
- **Validation Functions**:
  - `psX509ValidateGeneralName` should be used before calling `matrixValidateCerts` to filter out invalid input.

#### 2. JSSE (Java Secure Socket Extension)
- **Built-in Support**: SunJSSE [58] includes internal hostname checking (sun.security.util.HostnameChecker [42]).
- **Verification Capabilities**:
  - Supports domain name, IPv4, and IPv6 verification through the `HostnameChecker.match` interface.

#### 3. CPython SSL
- **Overview**: CPython is one of the oldest and most popular Python VM implementations.
- **SSL Support**:
  - Depends on the OpenSSL library but uses its own `match_hostname` function for hostname verification.
  - Currently supports domain name and IP address verification but does not support email verification.

#### 4. Apache HttpClient
- **Usage**: Widely used in Web-services middleware, such as Apache Axis 2.
- **Verification**:
  - Provides a `verify` function in `DefaultHostnameVerifier` for identity verification.
  - Can be enhanced with a `PublicSuffixMatcher` object for additional checks.

#### 5. cURL
- **Default Configuration**: Uses OpenSSL [7] but implements its own `verifyhost` function.
- **Verification Capabilities**:
  - Supports domain name, IPv4, and IPv6 verification.

### Developer Responses

#### GnuTLS
- **Patch Development**: Working on a patch to fix the issue of matching an IP address identifier in the subjectAltName when it is present in the CN [1].
- **Future Enhancements**:
  - Plans to provide a way to specify the identifier type to avoid confusion between hostnames and IP addresses [2].
  - Removing a fallback option that matches an IP address with a subjectAltName DNS [9], addressing potential attacks [3].
  - Recently introduced IDNA2008 support in version 3.5.9, enhancing DNS name format checks.

#### MbedTLS
- **Status**: Discussions are ongoing with the MbedTLS team regarding discovered issues.

#### MatrixSSL
- **Priorities**: Addressing RFC violations, including the incorrect order of checking between subject CN and subjectAltName (violation of RFC 6125) and case-insensitive matching of the local-part of an email address (violation of RFC 5280).
- **New Version**: Fixes deployed in version 3.9.0 [4], which also includes optional flags for hostname input validation and parameters for specifying the identifier type.

#### JSSE
- **Compliance**: The current version does not prioritize RFC 6125 compliance, but plans are in place to add this in future releases.

#### CPython SSL
- **Future Plans**: Deprecating their hostname verification implementation in favor of using OpenSSL's directly in the next release.

#### OpenSSL
- **Position**: Does not plan to address the issue of matching partial hostname suffixes in subject CN/subjectAltName, as it is not an RFC violation.
- **Other Discrepancies**: Believes that issues like matching wildcards in public suffixes or invalid hostnames should be handled at the application level or by certificate authorities.

#### HttpClient
- **Resolved Issues**: Fixed violations related to case-sensitive matching of subject CN (violation of RFC 6125 and RFC 5280) and attempting to match subject CN when subjectAltName is present (violation of RFC 6125) in version 4.5.3 (alpha release) [5].
- **Remaining Issues**: Other reported issues are considered correctly handled if the application uses `DefaultHostnameVerifier` with `PublicSuffixMatcher`.
