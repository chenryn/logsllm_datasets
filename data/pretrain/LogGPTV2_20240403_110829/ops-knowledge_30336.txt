### 问题描述

在使用OpenDHCPServer和OpenTFTPServer进行网络启动时，客户端在获取IP地址后开始下载引导文件，但中途停止了。具体错误信息如下：

**TFTP日志文件中的消息：**
```
[20-Aug-15 14:35:32] Client 10.0.0.3:2070 c:\tftp\pxelinux.0, Client 10.0.0.3:2070, Error Code 0 at Client, TFTP Aborted
[20-Aug-15 14:35:32] Client 10.0.0.3:2071 c:\tftp\pxelinux.0, 19 Blocks Served
```

根据我的了解，OpenTFTP默认的块大小为512字节。引导文件的大小应为53个块。我尝试传递一个只有1个块的假文件以期望捕获错误信息，但没有成功，客户端仍然停止（实际上光标仍在闪烁）。

### DHCP配置

```plaintext
[RANGE_SET]
DHCPRange=10.0.0.2-10.0.0.5
SubnetMask=255.255.255.0
BootFileName="pxelinux.0"
NextServer=10.0.0.1
```

### TFTP服务器配置

除了以下配置外，其他均为默认设置：

```plaintext
[HOME]
C:\tftp\
```

在正常服务中，IP分配和文件传输都能正常工作。

### 进一步调查

当我将最大块大小降低到512字节时，客户端成功传输了所有53个块。显然，服务器配置文件中的注释是不准确的。尽管如此，在此之后，客户端仍然停止。

经过进一步检查，发现问题是由于`pxelinux.0`文件导致客户端停止。而`grub4dos`则能正常加载。这是一个教训，不要盲目遵循教程。

### 解决方案

如果您的客户端在将块大小设置为512字节后才开始传输文件，这表明您的客户端不支持TFTP块大小选项RFC 2348。这意味着您的客户端PXE固件非常老旧且可能存在bug。建议您在尝试新的PXE启动之前升级固件。

### 总结

1. **确认客户端不支持TFTP块大小选项RFC 2348**：将块大小设置为512字节后，客户端能够正常传输文件。
2. **升级PXE固件**：确保客户端的PXE固件是最新的，以避免类似问题。
3. **检查引导文件**：确认`pxelinux.0`文件是否正确，并考虑使用其他引导加载程序如`grub4dos`。
