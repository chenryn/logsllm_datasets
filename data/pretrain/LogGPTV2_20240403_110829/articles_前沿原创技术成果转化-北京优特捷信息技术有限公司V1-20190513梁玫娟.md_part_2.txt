| Kib                                                                  |   |
| ana）相比处理性能得到十倍提升。开源的Elasticsearch为准实时索引检索引 |   |
| 擎，自研引擎则做到了真正的实时索引，内存原地检索不必消耗I/O，实时性  |   |
| 更高。传统的Elasticsearch打开索引时会加载较多的索引信息到内存中，如  |   |
| tip，dii                                                             |   |
| 等，这样打开索引慢                                                   |   |
| 且数量受限。在实际处理的场景中，需要保留的索引往往会达到半年以上，而 |   |
| 频繁查询使用的只有最近一两天的索引。自研引擎只在内存中加载少量的索引 |   |
| meta 信息，通过 LRU 的 cache                                         |   |
| 方式来管理索引文件信息，打开索引的内存占用小，因此可                 |   |
| 在保证低耗时的同时打开大量索引。在实际的机器数据处理中，平台大量使用 |   |
| ETL 的方式来进行日志的结构化。Elasticsearch                          |   |
| 不支持对字段类型的不兼容的字段进行索引，自研引擎对字段的索引方式和   |   |
| query                                                                |   |
| 的改写进行了专门的处理，以支持用户需求。而Elastic                    |   |
| search用于日志搜索这种对实时要求比较高的场景时，需要做性能调优，甚至 |   |
| 定制化开发，才能较好地满足大数据量、低延时的要求。自研引擎使用C++语  |   |
| 言开发，代码量控制在十多万行，也比40多万行的Elasticsearch大大精简。  |   |
