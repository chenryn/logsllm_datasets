一旦企业走到了这一步，就很难说服他们接受这样一个提议：“每台服务器上运行的软件售价 200 美元，并且您无法访问源代码，因此任何必要的修改都必须依赖我们。”这样的商业模式已成过去。我不确定在没有开源的情况下是否还能成功构建基础架构。开源还意味着拥有一个紧密团结的社区支持项目，所以我认为 Kubernetes 从一开始就占据了优势。

**Saron Yitbarek**：  
让我们回到容器编排工具的竞争中来。除了你提到的 Kubernetes，还有 Docker Swarm 和 Apache Mesos……

**Kelsey Hightower（00:17:00）**：  
当人们讨论容器编排竞赛时，我并不认为真正的竞争是在 Kubernetes、Mesos 或者 Docker Swarm 的用户之间展开的。实际上，这场竞争是为了争取那些尚未使用容器技术的潜在用户。这些用户可能还在使用传统的 Bash 脚本，不确定自己应该选择哪种编排工具。相比于已经选择了 Mesos 或 Swarm 的用户，这个市场的潜力要大得多。

**（00:17:30）**：  
这就是为什么容器战争会持续存在——关键在于如何帮助最终用户做出最佳选择。Mesos、Kubernetes 或 Docker Swarm 中哪一个将成为寻求更优解决方案用户的首选，目前还不确定（注：现已尘埃落定，Kubernetes 成为赢家）。但是，作为一个在这个领域工作的工程师，我可以告诉你，如果我们忽略市场营销和供应商因素，就会发现“不同的公司，同样的团队”。我们在彼此的基础上开发了许多工具，这些工具最终以某种形式出现在其他产品中。好主意就是好主意，不应该因为它们出自某个特定团队而被忽视。从技术和社区的角度来看，我们需要共享想法，同时也需要通过竞争来激发创新，让最好的点子脱颖而出，从而更好地满足用户需求。

**（00:18:00）**：  
因此，就这场竞赛而言，我们仍处于早期阶段，而且它本身并不是一项盈利活动。我们的目标不是直接向任何人销售产品，而是创建一个开放平台，让用户根据自身需求进行选择。我认为 Kubernetes 在这方面做得很好，它真正地开放并解决了实际问题。

**Saron Yitbarek**：  
听起来非常棒！我喜欢这种“同在一个球队”的概念。那么，对于容器、编排工具以及 Kubernetes 的未来，你有什么展望？

**Kelsey Hightower（00:19:00）**：  
我在 KubeCon 上做了一次主题演讲，提到了所有这些工具都非常出色，就像乐高积木一样。有了 Kubernetes，你可以自由选择用于安全或网络的产品，但作为开发者，你最关心的是能够轻松部署自己的代码。我相信 Kubernetes 和容器将成为底层技术，甚至是 Serverless 技术的基础平台。在未来，随着 Kubernetes 的普及，容器的应用将从个人和小规模供应商扩展到大型云服务提供商。尽管如此，容器可能会变得更加隐蔽，仅在后台默默工作，支撑着各种应用场景。

**Saron Yitbarek（00:20:00）**：  
Kelsey Hightower 是 Google 的员工开发人员。2017 年秋天，Docker 宣布支持 Kubernetes，但这并不意味着他们会放弃 Swarm；相反，他们决定与容器编排领域的明显胜者合作。Azure 和 AWS 也相继宣布了对 Kubernetes 的支持。与此同时，像 OpenShift 这样的 Kubernetes 发行版也在不断发展。这一切都表明 Kubernetes 核心正在变得越来越强大，能够适应新的用例，如微服务或持续集成项目。

**Clayton Coleman（00:21:00）**：  
这个生态系统最好能在类似于 Linux 的模式下发展，而我认为我们正朝着这个方向前进。优秀的开源项目总是能够让每个人都有机会参与进来，共同创造更好的成果。

**Saron Yitbarek（00:21:30）**：  
这一切都在迅速发生，毕竟这是一场竞赛。就在我们刚开始理解容器是什么的时候，第一轮比赛几乎就已经结束了。以下是来自 Red Hat 的 Scott McCarty 的观点。

**Scott McCarty**：  
回想两年前，容器镜像格式还是一个巨大的战场。然后，在六个月到一年前，容器编排成为了下一个焦点。紧接着，在 2017 年的 KubeCon 及其之前几周内，几乎所有主要供应商都宣布支持 Kubernetes。因此很明显，Kubernetes 已经赢得了这场战役。

**Saron Yitbarek（00:22:00）**：  
关于容器战争的故事即将结束，正如它的开始一样迅速。

**Scott McCarty**：  
现在，Kubernetes 已成为标准，这意味着应用定义也变得标准化了。任何人都可以在 YAML 文件中使用 Kubernetes 对象来定义应用程序，这是我们都希望看到的结果。事实上，我已经期待了 20 年，直到今天容器技术终于足够成熟，可以处理大规模系统。

**Saron Yitbarek（00:22:30）**：  
虽然 Kubernetes 的胜利似乎已是定局，但我们仍然面临着更大的问题。容器是否会成为未来几年的默认选择？它们是否会推动更多的云原生开发？这些变化又将催生哪些新工具和服务？以下是我们目前所知道的情况。

**（00:23:00）**：  
社区将继续通过 CNCF 改进 Kubernetes，并将其视为最重要的使命之一。我们将建立一套全新的容器技术。容器已经催生了大量的新型基础设施，随之而来的是对全新服务的需求。例如，Netflix 每周运行超过一百万个容器，这足以说明容器整合的程度和发展速度。毫不夸张地说，容器将是未来的基石。

**（00:23:30）**：  
整个季度里，我们一直在追踪开源运动的发展历程。首先见证了 Linux 如何主导市场，以及开源理念是如何改变商业、工作流程和日常使用的工具。容器无疑是开源运动中的一个重要里程碑，它们具有良好的可移植性、轻量化特性以及易于扩展的特点。

**（00:24:00）**：  
容器技术很好地体现了开源的优势，而开源项目也自然地推动了容器技术的进步。这是一个全新的世界，在这里，我们不再需要担心不同计算机或云环境之间的迁移障碍。

**（00:24:30）**：  
容器的标准化进程比预期更快。接下来的一集中，我们将转向另一场尚未解决的战争。这场云服务提供商之间的竞争前所未有地激烈，微软、阿里巴巴、谷歌和亚马逊四家巨头之间的摩擦正在升温。我们将跟随几位广受欢迎的代码英雄一起探讨这场“云端战争”。
