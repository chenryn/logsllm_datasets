**虚拟机逃逸**

iSight 固件的可重编程性可以被利用来实现虚拟机逃逸，从而使在客户操作系统中运行的恶意软件能够逃离虚拟机并影响主机操作系统。

一种方法是在虚拟机内部重新编程 iSight，使其充当 USB 人机接口设备（HID），例如鼠标或键盘。一旦 iSight 重新枚举，它将发送鼠标移动和点击或按键操作，主机操作系统会将其解释为用户的操作。

为了演示从 VirtualBox 虚拟机进行虚拟机逃逸的可行性，我们实现了一个 USB HID 键盘，加载后执行以下步骤：

1. 发送“主机键”按下；
2. 发送命令-空格键以打开 Spotlight；
3. 逐个发送“Terminal.app”的按键；
4. 等待几秒钟，发送回车键，并再等待几秒钟让终端打开；
5. 发送一个 shell 命令的按键，然后发送回车键；
6. 从 USB 总线断开连接，并修改其 USB 设备描述符以使用产品 ID 0x8300 ——这是 iSight 在未编程状态下的 PID；
7. 重新枚举。

VirtualBox 主机键默认为 Mac 主机上的左 Command 键，释放键盘所有权，使后续的按键操作传递给主机操作系统而不是客户操作系统 [19, 第一章]。

图 4 显示了一个从 VirtualBox 虚拟机内部重新编程的 iSight，正在向主机操作系统发送按键操作，指示 Spotlight 打开 Terminal.app。

当一个新的键盘首次插入计算机时，键盘设置助手会要求用户按下几个键以确定键盘布局。这种行为似乎由供应商 ID、产品 ID 和设备发布号控制。通过使用 Apple USB 键盘的适当值，助手不会出现，并且没有视觉提示表明操作系统认为已连接了新的键盘。输入到终端中的 shell 命令不受限制，例如，可以使用 curl 下载任意代码并运行。在 iSight 完成输入命令后，它会重新枚举为未编程的 iSight，这会导致 Apple USB 视频支持驱动程序用合法的 iSight 固件重新编程它，从而消除 iSight 是感染向量的证据。

尽管我们使用 iSight 进行虚拟机逃逸，理论上任何可以从虚拟机内部访问的 EZ-USB 设备都可以被重新编程为 HID 键盘。主要的限制是 USB 设备必须在攻击之前就已连接到虚拟机。默认情况下，虚拟机监视器不为此类设备提供连接，因此恶意软件需要诱使用户建立连接。即使设备已连接到虚拟机，固件也无法得知攻击是否按计划进行。它只能响应 USB 轮询发送按键操作。如果用户坐在计算机前，iSight 发送的按键可能会被注意到，用户可以通过打字或点击鼠标等操作进行干扰。一种部分补偿的方法是通过更改固件中的 USB 端点描述符来减少 USB 轮询间隔，从而使 iSight 能够更快地发送按键。

每个操作系统都有自己的策略来控制进程发送 USB 设备请求的能力。在 Linux 上，这由 udev 控制。如图 4 所示，我们在虚拟机内部使用 sudo 绕过客户操作系统的访问控制。或者，可以授予用户适当的权限。即使主机操作系统（OS X）对 USB 设备请求的使用没有任何限制，这些步骤之一也是必需的。由于每个客户操作系统在 USB 设备连接到虚拟机后控制对其的访问，要执行逃逸，恶意软件必须首先在客户操作系统中获得足够的权限以重新编程摄像头——这可能是一项非平凡的任务。
