#### 定义术语

**IdM 服务器和副本**

- 在 IdM 部署中安装第一个服务器时，必须使用 `ipa-server-install` 命令。随后，管理员可以利用 `ipa-replica-install` 命令在其他位置添加副本。默认情况下，这些副本会与创建它们的 IdM 服务器建立复制协议，从而实现数据同步。无论是初始服务器还是其副本，在功能上均无差异，都是全功能读/写 IdM 服务器。
- 已弃用名称：主服务器 (master server)

**IdM CA 续订服务器**

- 若 IdM 拓扑包括集成证书颁发机构(CA)，则一台服务器将承担 CA 续订服务器的角色，负责维护并更新系统证书。默认设置下，首个安装的 CA 服务器自动承担此职责，但任何 CA 服务器均可被配置为此角色。对于没有集成 CA 的环境，则不存在此类服务器。
- 已弃用名称：主 CA (master CA)

**IdM CRL 发布者服务器**

- 当 IdM 环境中存在集成 CA 时，一个特定服务器会被指定为 CRL（证书撤销列表）发布者，主要任务是管理和发布 CRL。通常，该角色由担任 CA 续订服务器的同一台机器承担，然而，用户可以选择任一 CA 服务器来执行这项工作。在缺乏集成 CA 的架构中，不设立 CRL 发布者。

**IdM 拓扑**

- 描述了 IdM 解决方案的整体布局，特别是各数据中心及集群间复制协议的设计。

**Kerberos 认证指示符**

- Kerberos 票据中的附加信息，反映获取票据所采用的原始验证方法：
  - `otp`：双因素认证（密码+一次性密码）
  - `radius`：RADIUS 协议下的远程身份验证
  - `pkinit`：基于公钥加密技术的身份验证
  - `hardened`：增强型密码保护机制，以抵御暴力破解尝试

**Kerberos keytab**

- Keytab 文件存储着 Kerberos 主体及其关联密钥的信息，使得服务能够检索自己的 Kerberos 密钥并进行身份验证。例如，每个 IdM 客户端都配备有 `/etc/krb5.keytab` 文件，记录了代表客户端计算机的 `host` 主体详情。

**Kerberos 主体**

- 表示 Kerberos 网域内每项实体（如用户、服务或主机）的唯一标识符，遵循如下命名规则：
  - 用户: `identifier@REALM`
  - 服务: `service/fully-qualified-hostname@REALM`
  - 主机: `host/fully-qualified-hostname@REALM`

**Kerberos 协议**

- 一种网络身份验证协议，通过加密技术为客户端和服务端应用程序提供强健的安全性保障。IdM 和 Active Directory 均依赖于 Kerberos 来完成用户、主机和服务的身份验证过程。

**Kerberos realm**

- 包含所有由 Kerberos KDC（密钥分发中心）管理的主体集合。在一个 IdM 实施环境中，Kerberos 域涵盖了所有的 IdM 用户、主机以及服务。

**Kerberos ticket 策略**

- KDC 通过连接策略实施访问控制，并通过生命周期策略管理 Kerberos 票据的有效期。例如，默认的全局票据有效时间为一天，而最大续订期限为一周。

**密钥分发中心 (KDC)**

- 作为 Kerberos 凭据信息的中央可信来源，KDC 负责签发 Kerberos 票据，并确保来自 IdM 网络内部实体的数据真实性。

**LDAP**

- 轻量级目录访问协议(LDAP)是一种开放且厂商无关的应用程序协议，用于通过网络访问和维护分布式目录信息服务。它基于 ISO X.500 标准定义的目录访问协议(DAP)，但更为简化。

**轻量级子 CA**

- 在 IdM 中，轻量级子 CA 是由 IdM 根 CA 或其下某 CA 签名的 CA，仅针对特定目的发放证书，比如保护 VPN 或 HTTP 连接的安全性。

**密码策略**

- 对 IdM 特定用户组设定的一系列条件，确保密码符合一定的安全标准，包括但不限于密码长度、字符种类数量及最长有效期等要求。

**POSIX 属性**

- 用于跨操作系统兼容性的用户属性，包括用户名 (`cn`)、账户名 (`uid`)、用户编号 (`uidNumber`)、主组号 (`gidNumber`) 及用户的家目录路径 (`homeDirectory`)。对于组而言，涉及组名 (`cn`) 和组号 (`gidNumber`)。

**复制协议**

- 同一 IdM 部署内两台 IdM 服务器之间的协议，保证数据和配置信息在两者之间持续同步。IdM 支持两种类型的复制协议：用于身份信息的“域复制”协议，以及用于证书信息的“证书复制”协议。

**智能卡**

- 一种可移动设备，内置集成电路芯片或其他形式的小型 USB 设备，允许用户将其插入主机计算机来进行身份验证。

**SSSD**

- 系统安全服务守护进程 (SSSD) 是 RHEL 主机上管理用户身份验证和授权的服务，可选择性地缓存从远程供应商获取的用户身份和凭证，支持离线验证。

**SSSD 后端**

- SSSD 后端或称数据提供商，是一个 SSSD 子进程，负责管理和填充 SSSD 缓存。它与 LDAP 服务器通信，执行查询并将结果保存至缓存中，同时处理在线身份验证请求，并应用相应的访问和密码策略。

**票据 (TGT)**

- 用户成功向 Kerberos KDC 进行身份验证后获得的一组临时凭证，称为票据授予票据 (TGT)。用户可以使用 TGT 请求其他服务的访问权限，从而实现单点登录体验。TGT 的续订限制及访问控制由 Kerberos 票据策略决定。

**Web 服务器**

- Web 服务器是接收并响应 Web 内容请求的软件和硬件组合，支持 HTTP 或 HTTPS 协议。红帽 IdM 使用 Apache Web 服务器来展示 IdM Web UI 并协调组件间的通信。

### 故障转移、负载均衡与高可用性

#### 客户端故障转移能力

- 默认情况下，IdM 客户端中的 SSSD 服务通过 DNS SRV 记录自动选择最佳 IdM 服务器进行连接。若当前服务器不可用，SSSD 将自动切换到另一个可用的 IdM 服务器。
- 若要出于性能考虑绕过 DNS 查询，可以从 `ipa_server` 参数中移除 `_srv_` 选项。
